## Deviations from Plan

### [Plan-allowed] D-03 dist-tag override applied to scripts.release

- **Found during:** Task 3.
- **Issue:** Pre-release mode in Changesets defaults to publishing under the dist-tag matching the pre-tag (`rc`). D-03 requires the dist-tag `next` for the RC ladder.
- **Fix:** Updated `package.json#scripts.release` from `... && changeset publish --provenance` to `... && changeset publish --provenance --tag next`. The plan explicitly authorized this path ("the `pnpm release` script SHOULD include `--tag next`").
- **Plan 05-07 follow-up flag:** If `changeset publish` silently ignores `--tag` (some Changesets versions do — RESEARCH Assumption A1-adjacent), the fallback is to bypass `changesets/action`'s publish step and run `npm publish --tag next --provenance` directly. Plan 05-07's first publish must verify which path actually applies the dist-tag.
- **Files modified:** `package.json`.
- **Commit:** `b070913` (bundled with the release.yml authoring commit).

### [Doc] TypeDoc warnings about decorator names in JSDoc text

- **Issue:** TypeDoc emitted 9 warnings parsing comments like `/** Phase 4 D-06: @Render shaper metadata */`. It interprets `@Render` as a block tag.
- **Severity:** Cosmetic. Build still produced 86 HTML pages and exited 0. No symbols were dropped from the output.
- **Fix:** None. To eliminate the warnings, the comments would need to escape the at-signs (`\@Render`) or rephrase. Not worth a code change in the publish-prep phase.
- **Tracked for:** Optional future cleanup; out of scope for v1.0.0-rc.1.

## Plan-05-07 Follow-up Checks

1. **Provenance pass-through (RESEARCH Assumption A1):** Confirm that `changeset publish --provenance --tag next` actually passes both `--provenance` and `--tag next` through to the underlying `npm publish` call when `changesets/action@v1` runs it. If either is dropped, switch the publish strategy to a direct `npm publish --tag next --provenance` outside of changesets/action.
2. **Provenance attestation:** After the first publish, verify on the npm registry page (or via `npm view @nirajk/express-controllers --json`) that the package has a `dist.attestations` object pointing at a Sigstore bundle.
3. **GitHub Pages enablement:** GitHub Pages must be configured (Settings → Pages → Source: gh-pages branch / root) before the first deploy run; the workflow assumes the branch will be created with `force_orphan:true` but the Pages site won't render until the repo's Pages source is set.
4. **Initial CHANGELOG.md creation:** `pnpm changeset version` (run by changesets/action's PR step) creates `CHANGELOG.md` on first version-bump. This file is not authored in this plan, per plan instruction.
