1. **Changesets** (D-08, DOCS-04) — initialize the CLI, configure for `access: 'public'` + `baseBranch: 'main'` + Keep-a-Changelog-style changelog, enter pre-release mode tagged `rc` (D-03), and seed the first changeset covering the Phase 1–4 surface (per RESEARCH "CHANGELOG seeding for 1.0.0-rc.1").

    Use the Write tool to create `.github/workflows/release.yml`. Content EXACTLY:

        name: Release

        on:
          push:
            branches: [main]

        concurrency:
          group: release-${{ github.workflow }}-${{ github.ref }}

        jobs:
          release:
            name: Release
            runs-on: ubuntu-latest
            permissions:
              contents: write       # changesets pushes version commits + tags
              id-token: write       # npm provenance via OIDC
              pull-requests: write  # changesets opens / updates the Version Packages PR
              pages: write          # gh-pages deploy
            steps:
              - uses: actions/checkout@v4
                with:
                  fetch-depth: 0

              - name: Set up pnpm
                uses: pnpm/action-setup@v6
                with:
                  version: 10

              - name: Set up Node.js 22
                uses: actions/setup-node@v4
                with:
                  node-version: 22
                  registry-url: 'https://registry.npmjs.org'
                  cache: 'pnpm'

              - name: Install dependencies
                run: pnpm install --frozen-lockfile

              - name: Single-package invariant
                run: bash scripts/check-single-package.sh

              - name: Create release PR or publish
                id: changesets
                uses: changesets/action@v1
                with:
                  publish: pnpm release
                  createGithubReleases: true
                  title: 'chore(release): version packages'
                  commit: 'chore(release): version packages'
                env:
                  GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
                  NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
                  NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}

              - name: Build TypeDoc
                if: steps.changesets.outputs.published == 'true'
                run: pnpm docs:build

              - name: Deploy docs to GitHub Pages
                if: steps.changesets.outputs.published == 'true'
                uses: peaceiris/actions-gh-pages@v4
                with:
                  github_token: ${{ secrets.GITHUB_TOKEN }}
                  publish_dir: ./docs
                  publish_branch: gh-pages
                  force_orphan: true

    Critical points:
    - `permissions.id-token: write` is the OIDC scope that lets `npm publish --provenance` mint an attestation (RESEARCH Pattern 7).
    - `permissions.contents: write` lets the changesets bot push the version commit + tags.
    - `permissions.pull-requests: write` lets the bot create the "Version Packages" PR.
    - `registry-url: 'https://registry.npmjs.org'` writes a correct `.npmrc` for `npm publish`.
    - `NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}` is the token actions/setup-node consumes; `NPM_TOKEN` is also passed for the changesets/action body (some versions expect both).
    - The `publish` input is `pnpm release` — which expands to `pnpm build && attw --pack . --profile node16 && publint && changeset publish --provenance` per Plan 05-02. CRITICAL: `--provenance` lives on `changeset publish`, NOT on a bare `npm publish` — Changesets passes the flag through (RESEARCH Assumption A1; verify in Plan 05-07's first publish — fall back if needed).
    - **D-03 dist-tag override (per checker revision 2026-05-10):** Changesets pre-mode defaults the published dist-tag to the pre-tag name (`rc`), but D-03 locks the dist-tag as `next`. To honor D-03, the `pnpm release` script SHOULD include `--tag next` on the `changeset publish` invocation (i.e. update `package.json#scripts.release` to `pnpm build && attw --pack . --profile node16 && publint && changeset publish --provenance --tag next`). If `changeset publish` ignores `--tag` (it does in some Changesets versions), the fallback for Plan 05-07 is to run `npm publish --tag next --provenance` directly — bypassing `changesets/action`'s publish step. Document in the Plan 05-06 SUMMARY which path was wired in `package.json#scripts.release` so Plan 05-07 knows what to expect.
    - Docs deploy happens ONLY when `steps.changesets.outputs.published == 'true'` — meaning the Version Packages PR was just merged and a publish ran. Pushes to main that DO NOT result in a publish (e.g., direct commits, the Version Packages PR creation itself) skip the docs deploy.
    - `force_orphan: true` keeps `gh-pages` clean (no growing history of every doc build).
    - The CI workflow (Plan 05-04) does NOT run on push to main when this workflow is also running — concurrency groups are separate, so they may run in parallel, which is fine.
