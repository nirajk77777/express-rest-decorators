<rules_locked>
The following decisions are LOCKED — do not revisit, ask the user, or expand scope:

1. **Scope:** `src/` only. NEVER touch `tests/`, `examples/`, `.planning/`, or root config files (`package.json`, `tsconfig*.json`, `biome.json`, `vitest.config.ts`, etc.).
2. **Removal rule:** Remove comments whose PRIMARY content is a GSD reference. Keep genuine WHY comments (invariants, workarounds, gotchas, non-obvious constraints) — but strip the GSD tag from them.
3. **JSDoc blocks:**
   - Strip phase tags inline when the JSDoc has other useful content.
   - If the ENTIRE JSDoc block is just a phase reference (e.g. `/** Phase 4 D-06. Pure registrar — no Reflect.defineMetadata (Phase 1 D-07). */`), DELETE the whole block. DO NOT fabricate replacement prose. Leaving public APIs undocumented is acceptable — a deliberate docs pass will fill gaps later.
4. **Barrel section headers in `src/index.ts`:** Keep as dividers but drop the phase tag. Example: `// Phase 2 — Express adapter (boot APIs)` → `// Express adapter (boot APIs)`.
5. **Stale forward-compat notes** (e.g. `/** Phase 3 — middleware classes/functions. Phase 2 accepts and ignores. */`): DELETE the whole JSDoc. The "Phase 2 accepts and ignores" half is now a lie post-Phase-3, and the rule says don't fabricate replacement docs.
6. **Cryptic fragments left after tag removal** (e.g. just `// arm 6`): DELETE them — they meant nothing without phase context.
7. **Do not write new JSDoc** to "fill" deleted blocks. Empty is acceptable.
8. **Do not do broader comment cleanup** ("what does this code do" rewrites). Out of scope.
</rules_locked>
